# Findings And Decisions

## Project Context

- Local path: `/Users/ngwa/Documents/欠薪社保计算平台`
- Current architecture: 单路由 React 客户端页面，计算、状态、导入导出和界面集中在 `app/page.tsx`。
- Persistence: `xinbao-rows`、`xinbao-double-rule`、`xinbao-meta` 三组 localStorage 数据，JSON 备份版本为 4。
- Current user flow: 用户先填写一个包含约 18 个字段的大表单，再生成逐月明细并手工修正。
- Current constraint: 无账号、无服务端业务数据、可直接推送 Git 仓库并静态化部署。

## LOCKED

1. 产品无需注册或登录，打开即可计算。
2. 业务数据默认只保存在当前浏览器，本产品不上传案例数据。
3. 总计必须包含欠薪、未续签双倍工资差额、社保公司尚欠补缴和公积金公司尚欠补缴。
4. 公司已经实际缴纳的社保、公积金金额必须保留并抵扣，只计算尚欠差额。
5. 合同期满后持续用工达到一个月时自动启用额外一倍工资，起算日为期满次日，累计上限为 11 个月。
6. 用户必须能逐月修正工资、实缴金额、基数、比例和异常月份，并能保存、备份和导出。
7. 降低普通用户输入数量和专业理解成本是本轮产品改造的首要目标。
8. TDD Iron Law：实现必须遵循 RED-GREEN-REFACTOR。
9. No Placeholders：需求、代码和交付中不得使用占位逻辑或含糊字段。
10. Verification Iron Law：没有针对行为的测试与成功构建证据，不得宣称完成。
11. 默认入口采用引导模式，完整逐月台账作为精算明细保留。
12. 首屏只展示入职日期、统计截止日期、合同月薪三个基础事实，再按用户选择的事项显示问题。
13. 连续实缴开始月等参与计算的推定值必须可见、可解释、可修改。
14. 结果先展示汇总和异常月份，完整精算台账默认折叠。
15. 本阶段不引入地区政策库；缺省最低比例必须显式展示来源边界并允许用户按当地规则修改，不得伪称为全国统一社保合计比例。
16. 社保、公积金问题优先收集公司实际每月缴纳金额；公积金默认最低单位比例 5%，社保以养老保险单位部分 16% 作为保守最低基线，并提示用户按参保地现行合计比例调高。
17. 合同开始日不参与未续签双倍工资计算，不得在引导流程中要求用户填写；只询问合同期满日，并用“合同上写的最后一天”解释。
18. 用户未填写社保或公积金缴费基数时，缺省测算基数取合同月薪；系统用“公司实际月缴金额 ÷ 缺省测算基数”反推实缴比例，采用比例取反推比例与用户确认的当地最低比例中的较高值，基数和最低比例均可手工修改。

## PROTECTED

1. 社保、公积金政策自动计算需要版本化的地区、年度和来源数据；首批支持范围需要用户确认。
2. 提供“粘贴缴费记录”入口，按月份和金额生成实缴明细，以覆盖断缴和金额变化场景。
3. 是否将自定义缴费口径保存为可复用本地预设。

## FLEXIBLE

- 引导流程使用单页步骤条、卡片问答或抽屉式交互均可，只要满足输入数量和状态合同。
- 具体视觉样式、图标、动画、组件拆分和 CSS 组织由实现者决定。
- 快捷比例可以使用 0%、30%、50%、100% 加自定义，也可以使用等价的单选控件。
- 在不改变存储语义和兼容性的前提下，可以拆分 `app/page.tsx` 中的计算与界面模块。
- 合同月薪使用“金额卡”展示，金额为视觉主体，币种和“元/月”为辅助信息，避免表格单元格式的硬分隔。

## Key Product Insight

用户愿意回答“我什么时候入职、工资多少、从哪月开始没发、最后缴到哪月”，但不应该被迫先理解“应缴基数、单位比例、连续区间、欠款状态”。产品应收集生活事实，再把专业字段作为可解释的推定结果展示。减少可见字段比单纯缩短表单更重要。

## Risks And Open Questions

- 政策自动计算需要地区、年度、基数上下限和公司比例数据；数据失效会造成比手动输入更严重的误导风险。实施前必须确认是否维护这套版本化政策数据。
- 官方依据：国务院办公厅《降低社会保险费率综合方案》明确养老保险单位缴费比例可降至 16%；公积金管理部门现行口径明确单位比例通常在 5%–12% 范围。社保其他险种费率仍由地区政策决定，因此不生成虚假的全国社保合计上限。
- “合同月薪等于缴费基数”只作为显式缺省测算口径；实际缴费基数可能受当地申报规则、上下限和经办机构核定影响。
- 连续实缴区间不能覆盖断缴、补缴和金额变化；必须保留粘贴记录或异常月编辑能力。
- 普通用户和专业用户的效率目标不同；引导模式不能删除现有精算明细能力。
- 当前 localStorage 与 JSON 备份需要无损迁移，不能因界面简化而丢失旧案例。
